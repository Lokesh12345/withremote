❯ Full-Stack Backend Assignment                                                 
  This assignment is backend-focused. There is no UI requirement; a CLI, a set  
  of scripts, or a couple of API endpoints tested with curl/Postman are all     
  acceptable. We care about how you handle data correctness and failure, not    
  how anything looks.                                                           
  Problem Statement 1: Can you build a sync pipeline that doesn't lie or        
  duplicate data?                                                               
  You're given 2-3 data sources that each expose records in a different shape:  
  a CRM, a payments processor, and a calendar/events feed. Each source supports 
  an incremental fetch (give it a cursor/timestamp, get back "what changed      
  since then") and a full fetch (get everything). Set up a free HubSpot         
  developer account for the CRM source, and the free Google Calendar API for    
  the events source, and seed each with a handful of sample records yourself.   
  Build a pipeline that ingests all three into one normalized schema despite    
  each source naming and shaping fields differently; that falls back to a full  
  backfill instead of silently losing data or crashing when an incremental      
  cursor goes stale or a source rejects it (e.g. a 410/expired-token error);    
  that writes idempotently, so the same webhook firing twice or the sync job    
  re-running back-to-back never produces duplicate rows; and that keeps going   
  when one source is down or returns garbage, so the other two still land their 
  data instead of the whole run wedging.                                        
                                                                                
                                                                                
                                                                                
  dont code analyse first the requeiment

  Core Objective

  Build a sync pipeline that ingests records from 3 heterogeneous sources into
  one normalized schema, while being resilient to the four classic
  distributed-sync failure modes. The grading criteria is explicit: "how you 
  handle data correctness and failure, not how anything looks." No UI.
  CLI/scripts/curl-tested endpoints all fine.

  The Three Sources

  Note: the assignment says "2-3 sources" but then names all three explicitly
  and says "seed each." Treat it as three. HubSpot and Google Calendar are
  mandated by name. The payments processor is unnamed — Stripe test mode is the
  obvious pick (best incremental/full API, webhooks, free).

  The Four Hard Requirements (this is what's actually graded)

  1. Normalization — each source names/shapes fields differently; map all into
  one schema. This forces a decision: what is the unified entity? HubSpot
  contacts, Stripe payments, and Calendar events are genuinely different nouns.
  You need either one generic record envelope (source, source_id, type,
  normalized common fields, plus raw payload) or per-type normalized tables. The
  generic envelope is more honest to "one normalized schema."

  2. Stale-cursor → full backfill fallback — when an incremental cursor is
  rejected (410 Gone, expired sync token, expired page token), do not lose data 
  or crash — fall back to a full fetch. This is the Google Calendar 410 Gone → 
  syncToken invalid case specifically (their API documents exactly this).
  HubSpot and Stripe use timestamp/ID cursors that don't expire the same way, so
  you may need to simulate a rejected cursor to prove the fallback works.

  3. Idempotent writes — same webhook fired twice, or the job re-run
  back-to-back, must never produce duplicate rows. This means an upsert on a 
  natural key (source + source_id), not blind inserts. Deterministic dedup key
  is the crux.

  4. Fault isolation — one source down / returning garbage must not wedge the
  run; the other two still land their data. Means per-source error boundaries,
  no shared transaction across sources, partial-success reporting.

  Hidden / implied requirements worth calling out

  - Cursor persistence — "incremental fetch since last cursor" means you must
  store each source's cursor durably between runs (a sync_state table). Where
  the cursor lives and how it's updated (only after successful write) is a
  correctness question.
  - Idempotency must survive the cursor too — you advance the cursor only after
  a batch is durably persisted, or you re-fetch and re-upsert (safe because
  upsert). Ordering of "write rows" vs "save cursor" matters for crash-safety.
  - Webhook path vs poll path — "same webhook firing twice" implies you may also
  expose a webhook receiver, not just a poll job. Both paths must funnel
  through the same idempotent upsert. An idempotency/event-dedup table for
  webhook delivery IDs is the clean approach.
  - "Garbage" handling — a source returning malformed records shouldn't kill the
  batch; per-record validation with a dead-letter/quarantine for bad records,
  not a crash.
  - Ordering / updates — if the same record changes twice, last-write-wins by
  source updated-at timestamp, so a stale re-delivery doesn't overwrite newer
  data.

  Key design decisions you'll need to make (I'd want your call on these before 
  designing)

  1. Language/runtime & storage — e.g. Python + Postgres/SQLite, or Node +
  Postgres. Upsert semantics (ON CONFLICT) matter, so a real SQL store beats a
  flat file.
  2. Payments processor — confirm Stripe test mode (my recommendation) vs mock.
  3. How real vs mocked — do you actually stand up HubSpot/Google/Stripe
  accounts and hit live APIs (assignment literally asks you to seed them), or
  build source adapters against a recorded/mock layer so it's reproducible? Best
  answer: real adapters, but with a fake/mock mode so the failure cases (410,
  garbage, downtime) can be triggered deterministically for the graders.
  4. Poll job only, or poll + webhook receiver — webhook receiver demonstrates
  the "webhook fired twice" requirement more convincingly.

  What "good" looks like for the grader

  - One normalized schema + a mapping layer per source.
  - A sync_state (cursors) table and idempotent upsert keyed on (source, 
  source_id).
  - Demonstrable proof of each failure mode: a test/script that fires the same
  event twice and shows one row; one that invalidates a cursor and shows the
  full-backfill fallback; one that kills a source mid-run and shows the other
  two still land.
  - Structured logging + a run summary (per source: fetched / written / skipped
  / failed).

  The normalized schema (the "one shape")

  A generic record envelope — more honest than three parallel tables, since the
  requirement says "one normalized schema":

  - records: id (surrogate), source (hubspot|stripe|gcal), source_id (natural id
  from the source), type (contact|payment|event), source_updated_at, title,
  email, amount, currency, status, occurred_at, raw_json (full original
  payload), synced_at.
  - Natural key = UNIQUE(source, source_id) → this is what makes upsert
  idempotent.
  - Last-write-wins guard: upsert only overwrites when incoming
  source_updated_at >= stored — so a stale re-delivery can't clobber newer data.

  Supporting tables:
  - sync_state: (source) → cursor, cursor_type, last_full_sync_at,
  last_success_at. The durable cursor store.
  - processed_events: (source, delivery_id) unique → webhook/event dedup ledger.
  Second firing of the same webhook is a no-op insert.
  - dead_letter: source, raw payload, error, timestamp — quarantine for garbage
  records instead of crashing.
  - run_log: per-run, per-source counts (fetched / upserted / skipped / 
  dead_lettered / failed, mode used, error) — the "doesn't lie" proof surface.

  How each of the 4 requirements is satisfied

  1. Normalization — each adapter owns a normalize(raw) -> Record mapping. Field

  2. Stale cursor → full backfill — each adapter's fetch_incremental catches the
  source's "cursor rejected" signal and raises a typed StaleCursorError; the
  orchestrator catches it, logs mode=full_backfill_fallback, calls fetch_full(),
  and resets sync_state. The canonical trigger is Google Calendar 410 Gone on 
  an invalid syncToken (real). HubSpot (timestamp) and Stripe (id cursor) don't
  expire naturally, so fake mode injects a rejection to prove the fallback path
  for all three.

  3. Idempotent writes — every write path (poll and webhook) goes through the
  same upsert(record) on UNIQUE(source, source_id). Re-running the job or
  double-firing a webhook produces zero duplicates. Webhook delivery double-fire
  is additionally short-circuited by processed_events. Cursor is advanced only 
  after the batch is durably committed — crash mid-run just re-fetches and
  re-upserts (safe).

  4. Fault isolation — orchestrator wraps each source in its own try/except and
  its own DB transaction. A source that's down, throws, or returns garbage
  records is confined: its bad records go to dead_letter, its failure is
  recorded in run_log, and the loop continues to the next source. No
  cross-source shared transaction.

  Fault/mock mode (how graders reproduce failures on demand)

  A --fake flag (or env) makes each adapter serve fixtures instead of live APIs,
  plus injectable faults via flags: --inject stale-cursor:gcal, --inject 
  down:stripe, --inject garbage:hubspot. This is what lets the three demo
  scripts run deterministically offline.

  plus --fake/--inject.
  5. webhook.py — receiver funneling into the same upsert.
  6. 3 proof scripts: demo_idempotency.sh (double-fire → 1 row),

  7. README.md — setup (real accounts + .env), how to run each demo, schema +

⏺ Before any code — here's my analysis of Problem Statement 2, how it relates to
  what we already built, and the decisions that need locking.

  What PS2 is really testing

  PS1 was about ingestion correctness. PS2 is about one number that can't drift
  — a single canonical revenue metric that stays identical no matter which
  endpoint asks, no matter how many sources or statuses get added. The grading
  is on definitional discipline, not features.

  Four distinct sub-problems, each with a trap:

  1. Canonical "collected" via allow-list, not exclusion list.
  Each source says collected differently (paid / succeeded / completed) and has
  non-collected states (pending, voided, refunded, failed). The trap the
  assignment explicitly calls out: an exclusion list (status NOT IN 
  ('refunded','failed')) silently lets a brand-new status like disputed or
  chargeback count as revenue. So the design must be:
  - A status registry: (source, raw_status) → canonical_status enum (COLLECTED /
  PENDING / REFUNDED / FAILED / VOIDED / UNKNOWN).
  - Revenue = rows whose canonical status is in the allow-list {COLLECTED}.
  - Any unmapped/unknown raw status maps to UNKNOWN → counts as $0, and is
  surfaced in a diagnostic so it doesn't vanish silently.

  2. Two views that always agree.
  A summary-total endpoint and a day/week breakdown endpoint. The invariant:
  summary(range) == sum(breakdown buckets in range), always. The only way to
  guarantee that is both endpoints call the same canonical function/SQL view —
  never compute revenue inline. Then a property test fuzzes random ranges and
  asserts equality.

  3. Invariance under new sources / new statuses.
  Adding a source or an unrecognized status must not change agreement and must
  not silently inflate revenue. This is the allow-list property again, plus:
  range semantics and timezone must be pinned identically in both views.

  4. "Catch a second divergent computation."
  If someone later writes SUM(amount) WHERE status='paid' somewhere else,
  something must fail. Enforcement layers I'd build:
  - One canonical module/SQL view is the only place revenue is defined.
  - A guard test that greps the codebase for status string-literals ('paid',
  'succeeded', SUM(amount) outside the canonical module and fails the build.
  - The agreement property test above.

  Hidden correctness requirements (drift sources)

  - Money as integer minor units or Decimal — never float. Float summation
  drifts.
  - Timezone pinned to UTC for bucket boundaries — otherwise summary (UTC) and
  breakdown (local) disagree at day edges.
  - Half-open ranges [start, end) consistently in both views — otherwise
  boundary rows double-count or drop.
  - Multi-currency: you cannot sum USD + EUR into one number honestly. Must be
  per-currency (or a single assumed currency).
